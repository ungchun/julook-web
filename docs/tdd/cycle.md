# TDD Cycle (Web 변형)

## 사이클

```
┌──────┐   ┌────────┐   ┌──────────┐   ┌─────────┐
│ RED  │ → │ GREEN  │ → │ REFACTOR │ → │ GREEN'  │
└──────┘   └────────┘   └──────────┘   └─────────┘
   ↑                                         │
   └─────────────────────────────────────────┘
```

- **RED**: 실패하는 테스트를 먼저 쓴다. 한 번 실행해서 **눈으로 실패를 확인**한다.
- **GREEN**: 그 테스트만 통과시키는 최소 구현.
- **REFACTOR**: 동작 보존, 구조 정리.
- **GREEN'**: 다시 모든 테스트 통과 확인.

한 사이클은 보통 30초 ~ 5분. 길어지면 단위가 너무 크다는 신호.

## 절대 규칙

> **테스트 없이 프로덕션 코드 한 줄도 추가/수정하지 않는다.**

예외 없음. 새 기능, 버그 수정, 리팩 모두 동일.

## "왜" — 이 사이클을 강제하는 이유

1. **테스트가 명세 역할**: 의도가 코드보다 먼저 표현됨
2. **회귀 방지 자동화**: 각 변경은 회귀 검증과 짝지어짐
3. **설계 압력**: 테스트하기 쉬운 코드 = 결합 낮은 코드
4. **iOS 본앱과 동일 철학**: 두 프로젝트가 같은 규율로 가야 마이그레이션/공유가 쉽다

## 도구 매핑

| 단계 | 명령 | 무엇을 보는가 |
|------|------|---------------|
| RED 검증 | `npm test -- --run path/to/file.test.ts` | 새 테스트가 빨갛게 뜸 |
| GREEN 검증 | 같은 명령 | 새 테스트 초록 |
| REFACTOR 검증 | `npm test` | 전체 초록 유지 |
| GREEN' 검증 | `npm run lint && npm run build && npm test` | 전체 통과 |

## 흔한 위반과 대응

| 위반 | 대응 |
|------|------|
| "테스트 나중에 쓰지 뭐" | 즉시 멈추고 RED부터 |
| "이건 너무 사소해서…" | 사소한 게 누적돼 회귀 만든다. 그래도 RED부터 |
| "타입만 바꾸는 건데?" | 타입도 동작이다. 테스트가 잡아내야 함 |
| "리팩 중인데 테스트 깨져서 잠깐 끄고…" | 끄지 말 것. 되돌리고 더 작게 쪼개라 |
